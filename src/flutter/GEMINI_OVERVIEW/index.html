Subsystem Overview

1. UI Layer (lib/features)

Components: ChatScreen, ServerSetupScreen

Role: Reactive views that bind to Riverpod providers. They display state and capture user intent but contain no business logic.

2. State Management Layer (lib/core/state & controllers)

Components: AppStateManager, SessionLifecycleController, ChatScreenController

Role: The orchestration layer. AppStateManager controls global app state (auth, server connection). Feature controllers (like ChatScreenController) bridge the UI to specific domain services.

3. Service & Domain Layer (lib/core/services & network)

Components: ConnectionRegistry, RoomSession, EventProcessor

Role: The core business logic.

  • ConnectionRegistry: A service locator that manages multiple server connections and RoomSession instances.
  • RoomSession: The "brain" of a chat room, maintaining message history and active state.
  • EventProcessor: A functional core component that strictly transforms state based on incoming server events (e.g., handling text deltas, tool calls).
4. Network & Infrastructure Layer (lib/infrastructure & core/network)

Components: NetworkTransportLayer, HttpTransport, Thread (from quick_agui)

Role: Handles physical communication. NetworkTransportLayer wraps HTTP and SSE clients, providing a single point for observability (NetworkInspector) and auth-refresh logic.

Key Data Flows

  • Initialization: AppStateManager initializes the ConnectionRegistry. When a server is selected, SessionLifecycleController ensures a RoomSession is created for the active room.
  • Outbound (Sending): User input moves from ChatScreen → ChatScreenController → RoomSession → Thread → HttpTransport → Server.
  • Inbound (Receiving): Server Events (SSE) flow from Server → NetworkTransportLayer → Thread → RoomSession. The RoomSession uses EventProcessor to calculate the new state, applying changes that propagate back up the stack to update the UI.
